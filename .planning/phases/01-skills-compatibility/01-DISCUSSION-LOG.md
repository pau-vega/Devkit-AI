## Cursor Skill Directory Strategy

| Option | Description | Selected |
|--------|-------------|----------|
| Real directories (copy SKILL.md into `.cursor/skills/<name>/`) | Simpler for end users who copy files manually; each plugin gets its own `.cursor/` tree | |
| **Symlinks** (` .cursor/skills/<name>/` → `skills/<name>/`) | Single source of truth; easier to maintain, but symlinks can confuse Windows users or manual file copies | ✓ |
| Build-script materialization | `build-marketplace.sh` copies skills into `.cursor/` during build; repo stays clean, but skill authors must remember to run the build | |

**User's choice:** Symlinks (single source of truth)
**Notes:** User requested research on how others manage this. Found Vercel Labs approach: symlinks generated by install scripts, not committed to repo. Build script creates them.

## YAML Frontmatter Fields

| Option | Description | Selected |
|--------|-------------|----------|
| **Minimal** (`name:` + `description:` only) | Satisfies OpenCode and Claude Code; Cursor also works | |
| **Enhanced** (`name:`, `description:`, `paths:` for TS/JS scoping) | Adds Cursor-specific file-type scoping (e.g., `paths: ["*.ts", "*.tsx"]` for typescript-rules) | |
| **Full** (`name:`, `description:`, `paths:`, `disable-model-invocation:` where needed) | Prepares skills for Cursor command-substitution in later phases | ✓ |

**User's choice:** Full (include `name`, `description`, `paths:` for Cursor scoping, `disable-model-invocation:` for later command-substitution)
**Notes:** All skills get `paths:` field with appropriate glob patterns.

## Symlink Strategy (Follow-up)

| Option | Description | Selected |
|--------|-------------|----------|
| **Committed to the repo** (`.cursor/` tree with symlinks checked into git) | Repo is self-contained; users who clone and copy `.cursor/` get working symlinks | |
| **Generated by `scripts/build-marketplace.sh`** | Symlinks created at build time; repo doesn't store `.cursor/` tree | ✓ |
| **Documented as manual step** | README tells users to create symlinks; repo doesn't store them | |

**User's choice:** Generated by `scripts/build-marketplace.sh` (Vercel Labs approach)
**Notes:** Don't commit `.cursor/` tree to the repo; let the build script create symlinks. Follows community practice (gitignore `.cursor/skills/`, generate at build time). Based on research: Vercel Labs uses symlinks generated by install scripts, not committed to repo.

## `paths:` Field Scope (Follow-up)

| Option | Description | Selected |
|--------|-------------|----------|
| **All skills** get `paths:` (e.g., typescript-conventions gets `paths: ["*.ts", "*.tsx"]`) | Consistent approach across all skills | ✓ |
| **Only TypeScript/JSDoc skills** (typescript-rules, jsdoc-standards) get `paths:`; workflow-toolkit skills omit it | Minimal, only where file-type scoping matters | |
| **Per-skill decision** | Analyze each skill and assign appropriate `paths:` or omit | |

**User's choice:** All skills get `paths:` field
**Notes:** typescript-conventions: `paths: ["*.ts", "*.tsx"]`; jsdoc-conventions: `paths: ["*.ts", "*.tsx", "*.js", "*.jsx"]`; workflow-toolkit skills: `paths:` appropriate to each skill's domain.

## Symlink Depth (Follow-up)

| Option | Description | Selected |
