# Research: Cursor + OpenCode Compatibility

## Executive Summary

**Cursor turns out to be a much closer fit for this marketplace than OpenCode** ŌĆö the opposite of what the existing 2026-05-05 research assumed by omission. Cursor ships a first-class hook system (`.cursor/hooks.json`) that is structurally a near-twin of Claude Code's `hooks.json`: `preToolUse` / `beforeShellExecution` / `beforeReadFile` / `afterFileEdit` events, command-based AND prompt-based hooks, exit code `2` to block, snake_case `tool_input.file_path` stdin payloads, plus a `matcher` field. Cursor also natively implements the **Agent Skills open standard** (`SKILL.md` with `name` / `description` / `paths` frontmatter at `.cursor/skills/<name>/` or `.agents/skills/<name>/`) ŌĆö the same format OpenCode uses and that Claude Code's existing skill files already match. Slash invocation via `/skill-name` works out of the box. Custom slash commands as a distinct concept do not exist in Cursor (Skills with `disable-model-invocation: true` are the substitute). Custom modes/agents are largely UI-driven and not file-portable in a marketplace-friendly way.

**Net effect for this project: Cursor parity is mostly free for skills and hooks**, modest for commands (translate to disable-invocation skills), and a real gap for sub-agent dispatch (no portable agent file format we can ship). **OpenCode parity is more invasive on hooks** (TypeScript rewrite, no shell script path) but easier on commands and agents (markdown with frontmatter). The previous OpenCode research is still substantively accurate ŌĆö the camelCase `output.args.filePath` field shape is confirmed by the official `.env` protection example in the current OpenCode plugins doc, validating the HOOK-01 question.

The biggest structural finding is that Cursor and Claude Code share an almost-identical `hooks.json` shape ŌĆö the existing seven shell scripts can be reused **verbatim** in Cursor by copying `hooks.json` to `.cursor/hooks.json` and adjusting matcher names (`Write|Edit` ŌåÆ `Write`, `Bash` ŌåÆ `Shell`). The same scripts cannot be reused in OpenCode because OpenCode requires a TypeScript plugin host, not a JSON config that spawns shell processes. This asymmetry should drive the roadmap: **Cursor is the cheap parallel runtime; OpenCode is the expensive one.**

The recommended ROADMAP shape is **option (a) ŌĆö integrate Cursor into existing Phases 1-4** rather than appending Phases 5-8. The work is not large enough to justify a doubled phase count and the per-phase tasks are highly parallel (e.g., the same SKILL.md frontmatter unlocks both runtimes simultaneously). Phases stay at 4 with two new sub-deliverables per phase ŌĆö a single Cursor track and a single OpenCode track ŌĆö closed out by one unified marketplace UI phase.

## Cursor Compatibility

### Rules / Skill Mapping

Cursor implements the **Agent Skills open standard** ŌĆö the same SKILL.md format used by OpenCode and conceptually compatible with Claude Code's skills. Because all three plugins already ship SKILL.md files that satisfy the spec (just need YAML frontmatter audited for `name` and `description`), **skills are zero-effort additive in Cursor.**

Cursor also has a separate `.cursor/rules/*.mdc` system for Always/Auto-Attached/Agent-Requested/Manual rules, but **this project should not target rules** ŌĆö skills are the better fit because:
- Skills support sub-references in `references/` (already used by `workflow-toolkit` and `jsdoc-standards`)
- Skills support glob scoping via `paths:` (matches the per-extension scoping needed for TS-only or JSDoc-only enforcement)
- Skills support manual invocation via `/skill-name` (substitute for slash commands ŌĆö see next section)

**Discovery paths Cursor scans (project-level):** `.cursor/skills/<name>/SKILL.md`, `.agents/skills/<name>/SKILL.md`, plus nested locations.

**Critical caveat:** Cursor does NOT scan `.claude/skills/` (only OpenCode does). This means each plugin needs either (a) a real `.cursor/skills/<name>/` directory, (b) a symlink from `.cursor/skills/<name>/` ŌåÆ existing `skills/<name>/`, or (c) the build script materializes a `.cursor/skills/` tree from the canonical `skills/` source. Option (b) is the cleanest in-repo authoring story; option (c) is friendlier for end-users who copy files manually.

### Commands

**Cursor does NOT support custom slash commands as a distinct construct.** The community forum and docs are explicit: `/migrate-to-skills` is a built-in command, but user-defined `/foo-command.md` files do not exist as a feature.

**Substitute:** Skills with `disable-model-invocation: true` in their frontmatter act as manual-only commands. Typing `/jsdoc-review` in Cursor chat invokes the SKILL.md content as if it were a slash command. This is the intended migration path and is what `/migrate-to-skills` automates.

This translates the three commands cleanly:
- `/ts-review` ŌåÆ `.cursor/skills/ts-review/SKILL.md` with `disable-model-invocation: true`
- `/jsdoc-review` ŌåÆ `.cursor/skills/jsdoc-review/SKILL.md` with `disable-model-invocation: true`
- `/create-workflow` ŌåÆ `.cursor/skills/create-workflow/SKILL.md` with `disable-model-invocation: true`

**Capability gap:** Cursor command-skills cannot dispatch a separate sub-agent (the `allowed-tools: [Agent]` Claude Code pattern). The skill body runs in the calling agent's context. For ts-review/jsdoc-review this is acceptable ŌĆö the body is just the review prompt and the main agent performs the review. The "structured Error/Warning/Suggestion report" output is preserved verbatim.

### Agents / Custom Modes

Cursor's "Custom Modes" feature is **UI-driven, not file-portable**. There is no documented `.cursor/modes/<name>.md` or equivalent that a marketplace plugin could ship. Modes are configured per-user in the Cursor UI and stored in user settings, not in the project repo.

**Practical consequence:** The `ts-reviewer` and `jsdoc-reviewer` agents have **no portable Cursor equivalent**. The roles those agents perform are absorbed into the command-skills (above) ŌĆö the skill body becomes the system prompt and the active agent executes it.

This is an honest capability gap. It does not block parity because the user-visible workflow (`/ts-review` returns a structured report) is preserved; only the internal architecture differs (no subagent dispatch, no isolated context).

### Hook Equivalent

**Cursor has a fully featured hook system that is structurally a near-clone of Claude Code's `hooks.json`.** This was the biggest single discovery in this research and substantially reduces the porting cost.

| Field | Claude Code `hooks.json` | Cursor `hooks.json` |
|---|---|---|
| Location | `<plugin>/hooks/hooks.json` | `.cursor/hooks.json` (project) or `~/.cursor/hooks.json` (user) |
| Schema root | `{"hooks": {"PreToolUse": [...]}}` | `{"version": 1, "hooks": {"preToolUse": [...]}}` |
| Tool match | `"matcher": "Write\|Edit"` (PascalCase) | `"matcher": "Write"` / `"Shell"` / regex on command (PascalCase confirmed for tool types) |
| Block signal | exit code 2 to stderr | exit code 2 OR `{"permission": "deny"}` JSON |
| Prompt hooks | `{"type": "prompt", "prompt": "..."}` | `{"type": "prompt", "prompt": "...", "timeout": 10}` |
| Stdin payload | `tool_input.file_path` (snake_case) | `tool_input.file_path` (snake_case, identical) |
| Plugin root var | `${CLAUDE_PLUGIN_ROOT}` | `${CURSOR_PROJECT_DIR}` env var (workspace root, not plugin install dir) |
| Fail-open default | yes | yes (set `"failClosed": true` to invert) |

**This means the seven existing shell scripts can be reused in Cursor with zero modifications.** The only required changes are:

1. Translate `${CLAUDE_PLUGIN_ROOT}/hooks/scripts/no-any.sh` ŌåÆ a script path resolvable from `.cursor/hooks.json` (e.g., `.cursor/hooks/no-any.sh` copied in, or a path under `${CURSOR_PROJECT_DIR}` if scripts live at a stable repo location).
2. Rename hook event keys: `PreToolUse` ŌåÆ `preToolUse`, matcher tool name `Bash` ŌåÆ `Shell` (Cursor's canonical name for shell tool calls).
3. Existing `"type": "prompt"` hooks port directly with the same prompt body and a small `"timeout"` addition.

**Field-name parity:** Cursor's `tool_input.file_path` matches the existing scripts' `jq -r '.tool_input.file_path'` extraction. No script rewrite required. This is the inverse of OpenCode where field names are camelCase and the entire script architecture is incompatible.

**Cursor hook events available** (relevant subset): `sessionStart`, `sessionEnd`, `preToolUse`, `postToolUse`, `postToolUseFailure`, `beforeShellExecution`, `afterShellExecution`, `beforeReadFile`, `afterFileEdit`, `beforeSubmitPrompt`, `stop`. Subagent events also exist (`subagentStart`, `subagentStop`).

**Subagent coverage:** Cursor exposes `subagentStart` / `subagentStop` events but the existing research does not yet validate whether `preToolUse` fires for subagent-driven tool calls in Cursor. This is the ONE empirical question worth validating live before shipping enforcement, paralleling OpenCode's #5894 known bypass.

### MCP Integration Path

Cursor supports MCP servers via `.cursor/mcp.json` (project) or `~/.cursor/mcp.json` (global) with stdio, SSE, and Streamable HTTP transports. The schema:

```json
{
  "mcpServers": {
    "ts-conventions": {
      "type": "stdio",
      "command": "npx",
      "args": ["-y", "ts-conventions-mcp"]
    }
  }
}
```

**Whether to ship an MCP server is a real architectural choice for this marketplace, but it is OUT of scope for v1 parity.** Reasoning:
- Hooks already cover the enforcement layer in both Cursor and Claude Code natively.
- An MCP server would duplicate logic already present in the shell scripts.
- MCP would only become attractive if we needed to expose enforcement to runtimes that have NO hook story (currently none ŌĆö both Cursor and OpenCode have hook equivalents).

**Defer MCP** to a future phase if/when a runtime appears that lacks both hooks and skills.

### Install / Discovery

Cursor has **no plugin marketplace and no `--plugin-dir` flag**. Like OpenCode, install is purely file-based:

```
1. Copy .cursor/hooks.json into your project root
2. Copy .cursor/skills/<name>/ folders into your project root
3. (Optional) Copy .cursor/mcp.json for MCP servers
4. Reload Cursor; auto-detection happens on save
```

A `<plugin>/.cursor/` directory inside each plugin folder (mirroring the proposed `<plugin>/.opencode/` structure) is the right packaging shape. The build script can produce a copy-paste install command for each plugin card.

**Comparable to OpenCode's `.opencode/` convention:** both runtimes treat hidden directories at the project root as their config namespace, with no manifest file. The marketplace landing page should explain both manual flows side by side.

### Field Schema Differences

Aggregated across all three runtimes:

| Concept | Claude Code | Cursor | OpenCode |
|---|---|---|---|
| Hook config file | `hooks.json` (PascalCase events) | `.cursor/hooks.json` (camelCase events, `version: 1`) | TypeScript plugin in `.opencode/plugins/*.ts` |
| Tool name in matcher | `"Write"`, `"Edit"`, `"Bash"` | `"Write"`, `"Read"`, `"Shell"`, `"MCP:<tool>"` | `"write"`, `"edit"`, `"bash"` (lowercase) |
| Stdin field | `tool_input.file_path` | `tool_input.file_path` | `output.args.filePath` (programmatic, not stdin) |
| Block | exit 2 | exit 2 OR `{"permission":"deny"}` | `throw new Error()` |
| Plugin root | `${CLAUDE_PLUGIN_ROOT}` | `${CURSOR_PROJECT_DIR}` (workspace, not plugin) | `ctx.directory` / `ctx.worktree` |
| Skill location | `<plugin>/skills/<name>/SKILL.md` | `.cursor/skills/<name>/SKILL.md` or `.agents/skills/` | `.opencode/skills/` or `.claude/skills/` (compat) |
| Skill frontmatter | `name`, `description` | `name`, `description`, `paths`, `disable-model-invocation` | `name`, `description`, `license`, `compatibility` |
| Slash command | `commands/*.md` with `allowed-tools` | Skill with `disable-model-invocation: true` (no separate concept) | `.opencode/commands/*.md` with `agent`, `subtask` |
| Subagent file | `agents/*.md` with `tools` array | UI-only Custom Modes (not file-portable) | `.opencode/agents/*.md` with `mode`, `permission` |
| Model field | `model: sonnet` | model picked at agent/UI level (no per-skill field) | `model: anthropic/claude-sonnet-4-5` (provider-qualified) |

**Critical note on stdin payload:** Claude Code and Cursor share `tool_input.*` (snake_case). OpenCode's hook is programmatic, not stdin-based, and uses `output.args.*` (camelCase). The proposed `jq -r '.tool_input.file_path // .tool_args.file_path // empty'` fallback in HOOK-04 is now unnecessary for OpenCode (which doesn't pipe stdin to shell scripts at all) ŌĆö keep `.tool_input.file_path` cleanly for both Claude Code and Cursor.

| Component | Claude Code (current) | OpenCode equivalent | Cursor equivalent | Approach |
|---|---|---|---|---|
| `skills/<name>/SKILL.md` | Reads natively from plugin dir | Reads from `.claude/skills/` or `.opencode/skills/` (auto-discovers) | Reads from `.cursor/skills/` or `.agents/skills/` (does NOT scan `.claude/skills/`) | **Additive copy/symlink.** Audit YAML frontmatter for `name`+`description`. Provide `.cursor/skills/<name>/` + ensure OpenCode finds `.claude/skills/` path (already true) |
| `commands/<name>.md` (e.g. `ts-review`) | `allowed-tools: [Agent]` slash command | `.opencode/commands/<name>.md` with `agent:` + `subtask:true` frontmatter | NO native custom commands ŌåÆ use Skill with `disable-model-invocation: true` | **Translation per runtime.** OpenCode: rewrite frontmatter, body verbatim. Cursor: become a skill, body verbatim. |
| `agents/<name>.md` (e.g. `ts-reviewer`) | `tools: ["Read","Glob","Grep","Bash"]` + `model: sonnet` | `.opencode/agents/<name>.md` with `mode: subagent` + `permission:` + provider-qualified model | UI-only Custom Modes ŌĆö NO portable file format | **Cursor: capability gap.** Inline the agent system prompt into the command-skill body. OpenCode: rewrite frontmatter, body verbatim. |
| `hooks/hooks.json` (PreToolUse + matcher + command) | Native | NOT supported ŌĆö silently ignored | Near-identical schema at `.cursor/hooks.json` | **Cursor: copy with key/matcher renames.** OpenCode: full TypeScript plugin rewrite (`.opencode/plugins/hooks.ts`). |
| `hooks/scripts/*.sh` (jq + exit 2) | Native (stdin tool_input.*) | Cannot reuse ŌĆö not invoked by OpenCode | Reused VERBATIM (Cursor stdin = `tool_input.*`, exit 2 blocks) | **Cursor: zero-edit reuse.** OpenCode: re-implement logic in TypeScript; can call shell via Bun `$` if desired but pure TS is cleaner. |
| `"type": "prompt"` inline LLM hook | Native | NO equivalent (#20387 unimplemented) ŌĆö capability gap | Native ŌĆö same `{"type":"prompt","prompt":"..."}` schema, add `timeout` | **Cursor: copy verbatim.** OpenCode: gap; covered by on-demand `/ts-review` skill instead. |
| `${CLAUDE_PLUGIN_ROOT}` variable | Native | `ctx.directory` / `ctx.worktree` in plugin code | `${CURSOR_PROJECT_DIR}` env var (workspace root, NOT plugin install dir) | **All three differ.** Cursor scripts must live at a path stable under workspace root (e.g., `.cursor/hooks/`). OpenCode resolves via plugin context. |
| `.claude-plugin/plugin.json` manifest | Native | None | None | **No translation needed.** Each runtime relies on directory convention. |

**Reuse density per component:**
- Skills: 100% reuse across all three runtimes (frontmatter audit only).
- Hooks (shell scripts): 100% reuse for Cursor; 0% reuse for OpenCode (TS rewrite).
- Hooks (config): 90% reuse for Cursor (key/matcher renames); 0% for OpenCode.
- Commands: 0% file reuse, 100% body reuse, frontmatter rewrite per runtime. Cursor merges them into skills.
- Agents: 0% file reuse for OpenCode (frontmatter rewrite, body verbatim); not portable for Cursor.

## Recommended ROADMAP Shape

**Recommendation: option (a) ŌĆö integrate Cursor into the existing 4 phases, not new phases 5-8.**

Reasoning:

1. **Skills work is the same one-time YAML frontmatter audit for both runtimes.** Phase 1 already adds `name`/`description` to all SKILL.md files; the same audit unlocks Cursor (`.cursor/skills/<name>/`) and OpenCode (`.claude/skills/<name>/` already scanned). Splitting this across two phases would create duplicate validation tasks.

2. **Cursor's hook port is dramatically cheaper than OpenCode's** (copy `hooks.json` + rename matchers vs. write a TypeScript plugin). Sequencing them in the same phase exploits this asymmetry ŌĆö Cursor enforcement ships in 1-2 days while OpenCode TypeScript work proceeds in parallel for ~1 week.

3. **Cursor commands collapse into skills**, which Phase 1 already produces. The `/ts-review` Cursor port is one frontmatter line (`disable-model-invocation: true`) added to the same SKILL.md content the agent's body uses. Adding a Phase 5 just for this is overhead.

4. **The marketplace UI phase (Phase 4) needs both runtimes' install instructions in one design pass.** A single redesign with three columns (Claude Code / OpenCode / Cursor) is cheaper than two separate redesigns and avoids visual asymmetry on plugin cards.

5. **Cursor parity exposes one new capability gap** (no portable subagent file) that doesn't exist for OpenCode. Documenting it inline alongside the OpenCode prompt-hook gap (Phase 3) is more coherent than splitting capability-gap docs across phases.

**Concrete phase deltas:**

| Phase | Existing scope | Add for Cursor |
|---|---|---|
| 1 ŌĆö Skills Compatibility | YAML frontmatter on 7 SKILL.md | Add `paths:` field where appropriate; create `.cursor/skills/<name>/` directories or symlinks per plugin (for ts/jsdoc only ŌĆö workflow-toolkit's 5 skills also get ported) |
| 2 ŌĆö Commands and Agents Port | `.opencode/commands/*.md` + `.opencode/agents/*.md` | Create `disable-model-invocation: true` Skill variants for `/ts-review`, `/jsdoc-review`, `/create-workflow` (Cursor has no agents ŌĆö inline reviewer prompt into skill body) |
| 3 ŌĆö Hook Enforcement Port | `.opencode/plugins/hooks.ts` TypeScript rewrite per plugin | Copy `hooks.json` ŌåÆ `.cursor/hooks.json`, rename `PreToolUse` ŌåÆ `preToolUse`, rename `Bash` matcher ŌåÆ `Shell`, copy 7 shell scripts into `.cursor/hooks/scripts/` (or fix paths). NO logic changes. Validate subagent coverage empirically (Cursor analog of #5894). |
| 4 ŌĆö Marketplace UI | Two-column install instructions (Claude / OpenCode), badges | Three-column install (Claude / OpenCode / Cursor), three badges per plugin card. `build-marketplace.sh` discovers `.cursor/` components in addition to `.opencode/`. |

**Effort delta vs OpenCode-only:** approximately +25-30% (Cursor adds tasks but most are very cheap reuse). Splitting into 8 phases would be ~+70% (full duplication of every phase including its own UI work).

**Capability gap matrix to surface in marketplace UI:**

| Capability | Claude | Cursor | OpenCode |
|---|---|---|---|
| Skills | full | full | full |
| Slash commands | full | via skill | full |
| Real-time hooks | full | full | full |
| Prompt-based LLM hooks | full | **full** (matches Claude) | gap (#20387) |
| Subagent dispatch | full | gap (UI-only modes) | full |
| Subagent hook coverage | full | unverified | gap (#5894) |

This is genuinely interesting marketing copy: **Cursor has prompt-based hooks (which OpenCode lacks); OpenCode has portable subagent files (which Cursor lacks). Claude Code has both.** The marketplace can position the plugins as runtime-aware rather than apologetic about gaps.
